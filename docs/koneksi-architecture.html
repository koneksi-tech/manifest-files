Koneksi platform architecture

Staging environment: GitOps-driven Kubernetes layout, application services, and backing data stores as defined in the manifest-files repository.

GitHub Actions · NCR · Argo CD · NHN NKS

0. Deployment strategy

Applications are built and published to NHN Cloud NCR (NHN Container Registry) by GitHub Actions in each service repository. Cluster state and image references are driven from the manifest-files Git repo. Argo CD on NHN Kubernetes Service (NKS) continuously reconciles those manifests; Argo CD Image Updater (optional) moves staging workloads to new NCR tags without editing Git.

CI pipeline (GitHub Actions)

Typical pattern per application repository: workflow triggers on push to main or on release tags → checkout → build container → authenticate to NCR → docker push with a version or -staging tag. NCR credentials are stored as GitHub encrypted secrets (not in this repo). The manifest repo does not build images; it only declares which image digest or tag workloads should run.

  • Pull: The cluster uses a docker-registry Secret (ncr-regcred) so nodes can pull private images from NCR.

  • Declarative deploy: Argo CD watches manifest-files; each Application points at a Kustomize path and destination namespace. Automated sync applies changes when Git (or Image Updater write-back) updates the desired state.
  • Image line: Staging Applications annotate argocd-image-updater.argoproj.io/image-list so base-image tracks an NCR repository; allowed tags often match ^v[0-9].*-staging$. write-back-method: argocd stores the new tag on the Application in the cluster.
  • Manual alternative: Bump the image in Kustomize or patches in Git and merge; Argo CD syncs on the next reconciliation.

1b. Argo CD setup (staging Applications)

Application manifests live under koneksi/argocd/environment/staging/. Each Application uses spec.source.repoURL git@github.com:koneksi-tech/manifest-files.git, targetRevision: main, and a Kustomize path below; destination is koneksi-staging. Register the Git repo in Argo CD first (argocd/SECRETS.md).

2. Kubernetes runtime (staging)

Application workloads run in koneksi-staging. Shared namespace and pull credentials are owned by the koneksi-backend Argo CD Application to avoid duplicate ownership warnings. Ingress exposes HTTP(S) via the cluster ingress controller.

Provenance pipeline; TimescaleDB for time-series; Mongo change streams / outbox where configured.

S3-facing gateway; uses existing ncr-regcred via imagePullSecrets; ingress host e.g. nhn-s3-gateway.koneksi.co.kr.

5. Security & operations notes

  • Repository credentials and TLS material should not be committed; use argocd/SECRETS.md patterns, Sealed Secrets, or external secret stores.
  • Restrict LoadBalancers with cloud security groups and loadBalancerSourceRanges where supported.
  • Rotate any credentials that were ever exposed in Git history.
